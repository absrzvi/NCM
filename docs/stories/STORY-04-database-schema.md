# Story: [STORY-04] Database Schema and Alembic Migrations
Status: READY
D-decisions touched: D8 (single Postgres container), D12 (draft_change_sets), D4 (idempotency_keys), D16 (parameter_history_cache)

## Why (from PRD)
The BFF owns all state in Postgres: drafts, audit events, idempotency keys, parameter history cache, user preferences, and environment configs (ARCHITECTURE.md §3 Data Model). Alembic manages schema migrations. Migrations run on BFF startup (not in CI — D8 single-container simplicity). This story creates all six tables and the Alembic migration framework.

## Assumptions (inherited from PRD + ARCHITECTURE)
- Postgres 16 runs as a single container on the same Docker Compose network.
- No read replicas, no failover, no operator (D8).
- Alembic migrations run automatically on BFF container startup via `alembic upgrade head` in the entrypoint script.
- All tables use UUID primary keys (generated by BFF, not Postgres).
- JSONB columns store structured data (drafts' `edits`, audit `detail`, history `history_json`, user preferences `policy_tree_collapsed`, environment `bench_allowlist`).
- No PII, no secrets, no hieradata values in any table — audit `detail` and draft `edits` store key_paths only, never actual values.
- Retention policies: audit 2y, idempotency 24h, history 5min, drafts 90d for SUBMITTED/DISCARDED, no TTL for ACTIVE or user_preferences or environment_configs.

## What to Build
1. `bff/alembic/` — Alembic migration framework:
   - `alembic.ini` — Alembic config pointing to BFF Postgres connection
   - `env.py` — Alembic environment script importing BFF models
   - `versions/001_initial_schema.py` — migration creating all six tables with indexes
2. Six Postgres tables (exact schema from ARCHITECTURE.md §3):
   - `audit_events` (id, created_at, fleet, puppet_environment, event_type, user_sub, user_role, detail, correlation_id, source; indexes on fleet+created_at, user_sub+created_at, event_type+created_at)
   - `idempotency_keys` (key, fingerprint, endpoint, user_sub, status_code, response_body, created_at, expires_at; PK on key+user_sub, index on expires_at)
   - `draft_change_sets` (id, fleet, puppet_environment, user_sub, status, created_at, updated_at, submitted_at, jira_issue, mr_url, branch_sha_at_creation, edits; PK on id, unique partial index on user_sub+fleet+status WHERE status='ACTIVE', index on fleet+status+created_at)
   - `parameter_history_cache` (id, cache_key, fleet, puppet_environment, branch, key_path, history_json, fetched_at, expires_at; PK on id, unique on cache_key, index on expires_at)
   - `user_preferences` (user_sub, last_fleet, last_puppet_environment, policy_tree_collapsed, updated_at; PK on user_sub)
   - `environment_configs` (fleet, puppet_environment, gitlab_project_id, gitlab_project_path, target_branch, layer_count, hiera_yaml_path, bench_allowlist, known_keys_path, active, created_at, updated_at; PK on fleet+puppet_environment, index on active)
3. BFF entrypoint script modification: `docker-entrypoint.sh` runs `alembic upgrade head` before `uvicorn bff.main:app`

## Acceptance Criteria
- [ ] Given Alembic is configured, when `alembic upgrade head` is run, then all six tables are created with correct schemas
- [ ] Given all tables exist, when `alembic downgrade base` is run, then all six tables are dropped
- [ ] Given the BFF container starts, when the entrypoint script runs, then `alembic upgrade head` completes before uvicorn starts
- [ ] Given two ACTIVE drafts for the same user and fleet are inserted, when the second insert is attempted, then a unique constraint violation occurs
- [ ] Given the `audit_events` table is queried with `ORDER BY created_at DESC`, when the index on (fleet, created_at DESC) exists, then the query plan uses the index
- [ ] Given the `idempotency_keys` table has a row with `expires_at` in the past, when a TTL sweep job runs, then the row is deletable (no foreign key constraints block it)
